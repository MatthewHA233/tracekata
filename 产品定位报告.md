# TraceKata 产品定位报告

> 用途：这份报告是为"专门思考产品定位"的新会话准备的。它汇总了客观的市场调研、同类竞品、核心竞争力判断、以及几轮对话沉淀下来的关键结论。**它不谈技术实现，只谈"这个产品到底是什么、该往哪走"。**
>
> 立场提醒：报告里的"洞察/结论"是分析和判断，不是定论。作者（用户 Maxwell）应带着批判去读，尤其第 5、6 节的取舍要自己拍板。竞品信息来自 2026 年的联网调研，可能随时间变化。

---

## 1. 产品现状（客观描述）

**一句话**：TraceKata 想帮"已经会用 AI 写代码、但读不懂/写不出/审不了代码"的人，通过"照着 AI 从真实项目降级出来的示范，手写誊抄 + AI 陪练"，把底层代码能力补回来。

**目标用户（作者自述）**：不是想脱离 AI 独立编程，而是想能——① 看懂代码；② 做简单封装（如给 AI 写 MCP 工具）；③ 审查 AI 写完的代码。

**当前形态**：Tauri 桌面端（React + Rust）。已实现的功能层层叠叠：
- 多项目管理（扫描含 `tracekata.json` 的目录）
- 渐进式练习"轨道"，`baseOffset`（"提前 5 个版本"用旧成品当基底复习）
- 每题三类文件：练习文件（自己敲）、示范文件（AI 写含小白注释）、题目索引（只给提示）
- 锚点 memo、知识库双向链接
- 内置 Claude Code AI 对话（流式、代理、多轮、读会话）
- （已删）VSCode 插件——曾负责编辑器内分栏

**核心动作**：**誊抄**——参考示范、手写、跑通、删掉、重来。

## 2. 市场调研：同类产品全景（2026）

### A. "照打真实代码"练手感 —— 成熟红海
- **typing.io**：最成熟。照打真实开源代码（jQuery、Rails、Linux kernel），记 WPM/准确率/问题键。
- **SpeedCoder**：类似，多语言，按难度进阶。
- **typelit.io**：照打文本/代码。
- **共性**：只练**打字速度/准确率**，不管理解、不管项目结构、不用 AI 生成、不做陪练。

### B. 交互式学习平台 —— 成熟红海（但机制不同）
- **Exercism、Codewars、freeCodeCamp、Codecademy、LeetCode、Advent of Code**。
- **共性**：让你**写代码解题**（从零解决问题），不是"照抄真实项目结构"，多数带 AI/导师反馈。

### C. "从 GitHub repo 生成学习内容" —— 拥挤红海
- **DeepWiki（Cognition/Devin）**：给任意公开 repo 自动生成 wiki + Q&A。
- **Greptile、Sourcegraph Cody、Onboard AI**：理解代码库、辅助 onboarding。
- **Pocket Flow "Tutorial-Codebase-Knowledge"**：把 repo 转成新手教程。
- **GitDiagram**：repo → 架构图。
- **共性**：都做**讲解 / 问答 / 画图**，**没有一个做"渐进式手抄练习"**。

### D. AI 代码审查工具 —— 成熟，但方向相反
- **CodeRabbit、Greptile、Graphite、Cursor Bugbot**：AI **替你**审查 PR。
- **关键**：它们**替人做**，某种意义上**加剧**了"人自己不会审"的退化。

### E. "AI 时代技能退化"话题 —— 真痛点，但无产品
- 2025–2026 广泛讨论（HN、Reddit r/ExperiencedDevs）："vibe code 得动，但从零写不出、读不懂、debug 不了"。
- 市场回应几乎全是**建议和手动 routine**：手抄 AI 输出、No-AI days、从零 rebuild 小项目、读源码、刷 kata。
- **没有一个成熟产品**把这件事产品化。

## 3. 红海 vs 空白（TraceKata 该躲开什么、该占什么）

| 领域 | 状态 | 对 TraceKata 的含义 |
|---|---|---|
| 照打代码练手感 | 红海（typing.io） | **别重复造**。若核心只是这个 = 造轮子 |
| 写代码解题 | 红海（Exercism 等） | 别碰 |
| 理解一个 repo | 红海（DeepWiki 等） | 别把自己做成"repo 讲解器" |
| AI 替你审查代码 | 红海（CodeRabbit 等） | 方向相反，别做替人审 |
| **AI 从真实项目生成"渐进手抄练习" + 陪练** | **几乎空白** | **潜在核心竞争力** |
| **练"人自己的"代码阅读 / 审查判断力** | **真空白** | **潜在核心竞争力，且对上作者真实目标** |

## 4. 几轮对话沉淀的关键洞察（诚实、可能刺耳）

### 洞察 1：软件的灵魂动作，不在软件里
TraceKata 的灵魂是"誊抄"，但誊抄发生在 **VSCode 里**。这个桌面软件做的所有事（选题、开文件、跑代码、聊天、记 memo、多项目管理、配色）都站在练习**外面**，是编排器/管理后台。**功能越多，壳越厚，离核心越远。** 作者隐约的不安来源于此。

### 洞察 2："做工具"可能是"练代码"的高级逃避
作者说要每天练 5 小时，但实际练到 014 就停下，大量时间花在**做这个工具**上。做工具有掌控感、即时反馈、宏大感；誊抄枯燥、暴露无能。**警惕：TraceKata 可能成了"感觉在为练习做准备"的体面拖延。**

### 洞察 3：核心是"脑"，不是"手"
作者的真实目标是**看懂 + 简单封装 + 审查 AI**——这是"脑"的能力（理解、判断），不是"手"的能力（打字速度）。**打字快 ≠ 会看会审。** 这一条直接决定了 typing.io 那类"手感"产品和 TraceKata 不是一个赛道。

### 洞察 4：作者其实已经在练目标能力了
在与 AI 结对做这个项目的过程中，作者不断读真实代码、审查 AI 输出、判断对错、叫停幻觉——**这正是"看懂 + 审查"的实战**，比誊抄练习更高强度。他可能低估了自己已经走到哪。→ 引申：对"看懂+审查"，**真实项目实战可能比降级誊抄更有效**。

### 洞察 5：做太复杂了（市场佐证）
活下来的同类产品都**极简专注到偏执**：typing.io 就一件事（照打+看数字），DeepWiki 就一件事（wiki+问答）。TraceKata 却背着多项目、轨道、baseOffset、三类文件、memo、知识库、AI 对话……**复杂度长在壳上，不在核心价值上。** 这是功能过载。

### 洞察 6：typing.io 引流的真相
"把手感练习外包给 typing.io"这个想法方向对（不重造轮子），但：
- 只能**引流**（放链接），不能**集成**（typing.io 不开放，喂不进你的练习内容）。
- 它练的"打字手感"和你的核心目标（看懂+审查）**几乎无关**。
- 所以它接手的是**你本来就不该背**的那部分。真正的减法是**放下"照打手感"这条线本身**，而不是把它外包。

## 5. 核心竞争力候选（二选一，需拍板）

市场空白落在两点，但 TraceKata 不该都要，要选一个当**灵魂动作**：

**候选 A：「AI 把真实项目降级成渐进手抄练习」**
- 差异化：抄的不是随机源码（typing.io），是 AI 从你指定的真实项目、按理解梯度降级出来的有意义结构。
- 风险：本质仍是"手抄"，离"打字手感"近，可能练成"能抄不能创/不能审"；且和洞察 3（核心是脑不是手）有张力。

**候选 B：「练人自己的『看懂 + 审查』判断力」**
- 差异化：市场全在"替人做"（替你写/审/懂），这个逆行——**让人自己变强**。真空白，且正对作者真实目标。
- 形态想象：AI 给一段真实/它写的代码 → 你判断它对不对、有没有坑、意图是否达成、动手改 → AI 给反馈。像"代码审查的健身房"。
- 风险：更难设计成一个爽的核心动作；没有现成范式可抄。

**报告的倾向（供参考，非定论）**：候选 B 更契合作者真实目标、更逆行市场主流、想象空间更大；候选 A 更容易做但更可能沦为 typing.io 的次等替代。

## 6. 往哪发力（行动建议）

1. **先做减法，别再加功能。** 用一把尺子审视每个功能：*它让核心动作发生得更多，还是只是让软件更"完整/好看"？* 后者一律砍成可选或删除。多项目管理、置顶归档、知识库双链、配色——大概率是壳。
2. **确定唯一核心动作**（候选 A 或 B），把它做到 typing.io 那种偏执程度，其余全是配角。
3. **亲手体验竞品**：花 20 分钟真用一次 **typing.io**，体会"极简专注"和"纯照打不管理解"到底够不够——这会给"核心动作该是什么"最直接的身体感觉。
4. **先当第 1 个真实用户，再当作者**：用现在的半成品，连续两周每天真练下去。若两周后还在改配色加功能而没真练 → 洞察 2 被验证，产品的灵魂需要重新面对。

## 7. 留给新会话拍板的关键问题

1. **核心动作是"手"还是"脑"？**（候选 A vs B）——这是最根本的一题，定了它，其余都好办。
2. **谁是最精确的目标用户？** "会 vibe coding 但想补底层能力的人"太宽。是转码新人？是资深但被 AI 惯坏的人？是想给 AI 写工具的产品经理型开发者？画像越窄越好。
3. **它是个产品，还是作者的个人工具？** 若只是让作者自己每天动手的私人仪式工具，那"有没有人用"不重要，标准是"它改变了作者的行为没有"——这也完全合法。先诚实回答这个，再决定投入。
4. **减到什么程度算够简？** 想象 v1 只有一个屏幕，那个屏幕是什么？

## 8. 可参考的具体产品（去亲手看）

- **typing.io** —— 极简专注的标杆；照打真实代码练手感。
- **SpeedCoder** —— 同类，多语言进阶。
- **Exercism** —— "刻意练习"叙事做得好的学习平台（写代码解题）。
- **DeepWiki**（deepwiki.com）—— repo → wiki + 问答，看"从真实项目生成内容"的成熟做法。
- **Pocket Flow / Tutorial-Codebase-Knowledge** —— repo → 新手教程，最接近"从 repo 生成学习内容"。
- **CodeRabbit / Cursor Bugbot** —— AI 代码审查（反面参照：它们替你审，你要做的是让人自己会审）。

## 9. 一句话总结

TraceKata 站在一个**真实且几乎无人做好的缝隙**里（AI 时代帮人**自己**变强，而非替人做），所以不是闭门造车。但它**做得太复杂**，复杂度长在壳上不在核心；而且它的核心该是"脑"（看懂+审查）不是"手"（打字），这决定了它和 typing.io 那类产品不是一个赛道。**下一步不是加功能，是想清楚唯一的核心动作、狠狠做减法、并先让作者自己真正用起来。**
